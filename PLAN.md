# Plan: Remove the legacy single-pair merge path

_Locked via grill — by Claude + kiddulu916. (Deferred from candidate #2's grill; scope confirmed.)_

## Goal

`GameCubit.merge()` (`game_cubit.dart:269-326`) is a fully-wired production method — undo frame, golden bonus, drop, persistence — that no UI can reach (`BoardWidget` only emits chains) and whose output the server can never accept: TS `parseEvent` (`engine.ts:246-254`) parses only `"chain"`/continue events, so a log containing a `MergeEvent` is rejected, a fact permanently pinned by the `reject-legacy-merge-event` golden sentinel. Delete the dead path: `GameCubit.merge()`, `GameEngine.merge()`, and `GameEngine.goldenBonusFor` (pair-form, orphaned by the removal). KEEP `MergeEvent` (in `move.dart`, with its `"merge"` parse case) and `GameEngine.canMerge` — the golden-vector rejection sentinel constructs typed legacy logs with both (`golden_vectors_test.dart:258,297`), and that proof must outlive the code it guards against. Migrate the merge-based tests that pin unique behavior to `playChain`; delete the ones that only exercised the removed API. No TS change, no season bump — accepted-log semantics are untouched (the server was already chain-only).

## Key decisions & tradeoffs

- **`MergeEvent` + `canMerge` survive** (grill decision, carried from candidate #2): the rejection sentinel is the permanent, executable proof that the server rejects legacy logs. Rewriting it to raw JSON to purge the types would rework the generator and both suites to prove the same thing with less type safety — rejected. Codex proposed deleting `canMerge` too (its only use is the sentinel GENERATOR, which could use `isValidChain`); modified rather than adopted: the user explicitly chose to keep it, so it stays with its doc corrected to name the generator as its sole consumer — the false "chain-validator building block" claim is dropped. The TS suite replays the committed sentinel and asserts rejection on every CI run, so the sentinel's proof-value never depends on `canMerge` executing in CI.
- **`goldenBonusFor` goes with `merge()`**: its only production caller is the removed method; the chain path computes the same per-golden-tile bonus inline (3 lines). Keeping a tested-but-uncalled helper is dead code with a maintenance face.
- **Migrate-or-delete per invariant, not per test**: the undo suite's chain-based coverage already exists; merge-based duplicates die, unique invariants (golden-coin refund, PRNG rewind) live on as chain tests. Coverage is measured by invariants pinned, not test count.
- **No season bump** (settled from code): `kLeaderboardSeason` fences scoring-rule changes; this PR changes what the CLIENT can emit, not what the server accepts — the server rejected merges before and after.
- **No TS-side change at all**: `engine.ts`/`constants.ts` untouched; the deployed bundle stays v8. (No redeploy needed — first candidate since #2 to leave `_shared/` alone.)

## Risks / open questions

- The undo tests' merge-based variants may encode subtly different expectations (single-pair merge scored differently than a 2-chain); migration must re-derive expected values for the chain equivalents rather than copying numbers — the red-green discipline in the docs plan catches this.
- `game_state.dart` or other files may reference `merge` in doc comments only — sweep and update prose, no behavior.
- If any test drives `GameCubit.merge()` to produce a snapshot that OTHER tests then load, deletion order matters; the per-task red-green sequence keeps the suite green throughout.

## Out of scope

- No changes to `MergeEvent`, `move.dart` parsing, `canMerge`, the golden-vector fixture, or either golden-vector test suite.
- No TS mirror changes, no deploy, no season bump.
- No changes to `playChain`, undo mechanics, scoring, or any live rule.
- The `merge()` name on `Cubit`/bloc internals or Dart collection methods — only the two `GameEngine`/`GameCubit` methods and the orphaned helper.
