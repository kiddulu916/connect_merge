# Plan: Golden-vector fixture for the dual-engine seam

_Locked via grill — by Claude + kiddulu916. Revised after Codex rounds 1–2._

## Goal

Make the Dart↔TS engine-parity invariant executable at the **source** level. Commit a single JSON fixture of recorded runs — `(date, difficulty, moveLog) → expected outcome` — generated by the real Dart client path, asserted by both the Flutter test suite and the Deno test suite, and run automatically by a new minimal GitHub Actions workflow. Source drift between the engines then fails a red PR check instead of shipping and rejecting legitimate player scores. (Deployed-bundle drift — code merged but `submit-score` not redeployed — is a separate operational gap, explicitly out of scope by user decision; see Out of scope.)

## Approach

1. **Fixture file**: `supabase/functions/_shared/golden_vectors.json`.
   - Shape:
     ```json
     {
       "_readme": "<regeneration policy>",
       "season": <kLeaderboardSeason>,
       "baseDate": "<fixed UTC date the generator starts from>",
       "vectors": [ { "name", "date", "difficulty", "rule": "<challenge vectors only>",
                      "moveLog": [...MoveEvent.toJson...],
                      "expected": { "valid": true, "score", "highestTier", "status" } } ],
       "rejections": [ { "name", "date", "difficulty", "rule?", "moveLog",
                         "expected": { "valid": false } } ]
     }
     ```
   - No generation timestamp — regeneration is byte-for-byte deterministic; `baseDate` is the only date anchor.
   - `_readme` states the policy: regenerate ONLY on an intentional rule change, in the same commit as the `kLeaderboardSeason` bump in both languages.
   - **Both suites assert `season` equals their own `kLeaderboardSeason`** — this also cross-checks that `constants.dart` and `constants.ts` agree on the season.
   - **Coverage contract is hardcoded in both tests, not in the fixture**: each suite carries the same required scenario-name set (honest AND rejection names) and asserts exact set equality with the fixture's contents. Deleting a scenario from the fixture fails both suites; the fixture cannot self-validate its coverage away.
   - Location rationale: Deno JSON-imports it relatively (`import vectors from "./golden_vectors.json" with { type: "json" }` — no `--allow-read`, documented test command unchanged); Dart reads `File('supabase/functions/_shared/golden_vectors.json')` (flutter test cwd = repo root); not imported by any `index.ts` so it never ships in an Edge Function bundle.

2. **Generator = the Dart assertion test**, env-var guarded: `test/domain/engine/golden_vectors_test.dart`.
   - Normal run: loads the fixture, replays each honest vector through `GameCubit` (the real client scoring path — no third engine), asserts final `board.score`, `highestTier`, and `status` match `expected`.
   - With `UPDATE_GOLDENS=1` in the environment: drives scripted runs and rewrites the fixture; fails (with the missing scenario names) if any required scenario can't be produced.
   - **Season diff-guard**: before writing, the generator diffs the **complete semantic payload** — the full `vectors` and `rejections` arrays (dates, rules, move logs, expected values; everything except `_readme`/`season`) — against the committed fixture. If anything changed while `season` is unchanged, it refuses with an instructive error ("rules changed without a season bump — bump kLeaderboardSeason in BOTH constants files, or set UPDATE_GOLDENS_FORCE=1 if this is a policy-only regeneration"). The force flag exists because changing the *play policy* legitimately changes vectors without any rule change.
   - Cubit harness: `InMemoryStorageService`, no-op hooks. Golden-tile coin bonuses don't touch `score`, so they can't contaminate vectors. Ad continues are granted only after checking `canOfferAd` — mirroring the UI, since `grantAdReward()` itself does not guard the cap (game_cubit.dart:638).
   - Play policy (deterministic, bounded — no exponential search): scan cells in index order; play the **first legal chain** found. Chain-length preference is per-rule: 2-tile default; **first legal 3-tile chain under `longChainsOnly` AND `comboRush`** (`comboRushMultiplier(2) == comboMultiplier(2)`, so only a 3+ chain distinguishes the rule). Hard per-move search cap. If no rule-legal chain exists while status is `playing`, end the run there — a prefix move log is a valid, verifiable run.

3. **Honest-vector matrix** (enforced by the hardcoded coverage sets):
   - All 4 standard difficulties (easy, medium, hard, legendary) on `baseDate`.
   - All 6 challenge rules — search dates in `[baseDate, baseDate + 60 days)`; fail generation naming any rule not found. Each challenge vector stores its `rule`, and **both suites assert their own `challengeRule(date)` derivation equals the stored rule before replaying** — pinning rule derivation itself cross-language.
   - **Distinguishing-move requirements** (generator fails if unmet, advancing to the next matching date in the window): the `comboRush` vector must contain ≥1 chain of length ≥3; the `longChainsOnly` vector must contain ≥1 accepted 3-chain (a non-empty, rule-distinguishing run).
   - One run that exhausts moves and uses **all three** ad-continues (`kMaxAdContinuesPerDay = 3`), exercising `kAdMoveReward` and the full continue budget.
   - One run ending `outOfMoves` at full budget with no continues.
   - Deadlock ending: **excluded entirely** (refill guarantees a merge unless the board is 100% full, so it's likely unreachable by the scripted policy, and a "bonus" vector would break the exact set-equality check). Deadlock behavior stays covered by the existing hand-written unit tests in both suites.

4. **Rejection vectors** (TS-asserted only — the Dart cubit ignores illegal moves rather than rejecting runs). Only cases **not already covered** by `engine.test.ts`'s hand-written rejection tests. Each sentinel is **constructed from a known Dart board state (terminal or mid-run, as the case requires), and the generator asserts every precondition except the targeted guard holds** — so the rejection can only fire for the reason under test:
   - A **fourth** ad-continue appended to the honest all-three-continues vector — generator asserts the terminal board is `outOfMoves` **with a merge still available**, so only the cap can reject it.
   - A chain appended after the run reaches `outOfMoves` — generator asserts the path is geometrically legal on the terminal board, so only the move budget can reject it (pins the standard `kMovesPerDay` budget cross-language; the cheap substitute for exposing `movesRemaining` in `VerifyResult`).
   - The same post-budget sentinel for **`budgetCut`** (pins `kChallengeMoves = 15`) and for **one non-budget challenge rule** (pins that other challenges get `kMovesPerDay = 30`) — without these, TS could grant challenge runs 16 or 31 moves and still accept every honest prefix.
   - A **2-chain under `longChainsOnly`** appended to that vector's log at a point where it's otherwise legal — pins the rule's length gate cross-language.
   - Legacy `{"type":"merge"}` event inserted **mid-run on a `playing` board** with an otherwise-legal from/to pair (not appended to a terminal board, where the rejection would be ambiguous) — pins that the dead `MergeEvent` wire form itself stays server-rejected.
   - Existing `engine.test.ts` rejection cases (illegal geometry, malformed events, etc.) stay where they are — no duplication.

5. **Deno assertion test**: new `supabase/functions/_shared/golden_vectors.test.ts` — JSON-import the fixture; assert `season`; assert the hardcoded coverage set matches; for challenge vectors assert `challengeRule(date)` equals the stored `rule`; run every honest vector through `verifyRun`/`verifyRunChallenge` and assert `{valid: true, score, highestTier}` exactly (`status` is Dart-side only — `VerifyResult` doesn't expose it, and changing production types is out of scope); run every rejection vector and assert `valid: false`. Existing `engine.test.ts` stays untouched — it pins PRNG/seeder internals the fixture doesn't.

6. **Minimal CI**: `.github/workflows/test.yml` — on push/PR to `main`:
   - `permissions: contents: read`; third-party actions pinned by commit SHA; Flutter and Deno versions pinned explicitly (with pub cache).
   - Job A: `flutter analyze` + `flutter test`. Job B: `deno test --frozen supabase/functions/` (the repo has a tracked `deno.lock`; `--frozen` enforces it).
   - No secrets, no deploy steps.
   - Documented manual follow-up (GitHub settings, not repo files): mark both jobs as required status checks on `main` so a red run actually blocks merging.

7. **Docs**: per the repo's planning workflow, mirror this plan as `docs/superpowers/specs/2026-07-14-golden-vectors-design.md` + a task plan under `docs/superpowers/plans/`; add the golden-vector commands (assert + `UPDATE_GOLDENS=1` regenerate + `UPDATE_GOLDENS_FORCE=1` escape hatch) and a pointer to the fixture in **both** `CLAUDE.md` and `AGENTS.md`.

## Key decisions & tradeoffs

- **Dart replay vehicle is `GameCubit`, not a new Dart `verifyRun`** — a Dart verifier would be a third engine to keep in sync; the cubit is the code path that actually produces prod scores. Cost: the Dart test is heavier and depends on cubit orchestration (refill loop lives there until candidate #2 moves it).
- **No production code changes, period** — `VerifyResult` is not extended (budget drift is pinned by the chain-after-outOfMoves rejection sentinel instead), and `grantAdReward()`'s missing internal guard is worked around in the generator (mirroring the UI), not fixed here. Both are candidate-#2-adjacent cleanups.
- **One JSON fixture, JSON-imported by Deno** — avoids `--allow-read` changes and twin-fixture drift. Cost: fixture lives under `supabase/functions/_shared/` rather than a neutral `test/fixtures/`.
- **Generator is env-guarded inside the assertion test** — one file, one documented command, mirrors Flutter's `--update-goldens` idiom.
- **Coverage sets hardcoded in both tests** (round-2 change, replacing the fixture-embedded manifest) — a fixture key can be edited in the same commit that deletes a vector; a hardcoded set in each suite means shrinking coverage requires visibly editing tests in two languages.
- **Season diff-guard in the generator** (round-2 change, upgrading round 1's assertion-only stance) — the plain `season == kLeaderboardSeason` assertion can't catch "regenerated after a rule change without bumping season," because the generator writes the current constant. The diff-guard catches exactly that; `UPDATE_GOLDENS_FORCE=1` covers legitimate policy-only regenerations.
- **First-legal-chain policy, not longest-chain** — longest simple path is exponential; parity testing doesn't need interesting play, it needs deterministic, bounded, reproducible play. Per-rule 3-chain preference exists solely so `comboRush`/`longChainsOnly` vectors actually distinguish their rules.
- **Vectors pin final outcome only, not intermediate boards** — keeps the fixture small; intermediate-state parity is already covered by `engine.test.ts`'s hand-written board vectors.

## Risks / open questions

- Under `longChainsOnly`, runs may end early (no legal 3-chain) — mitigated by the distinguishing-move requirement: the generator advances to another qualifying date rather than accepting an empty run.
- Challenge-rule date search assumes all 6 rules appear within 60 days (uniform PRNG over 6 variants: P(miss) ≈ 6×(5/6)^60 ≈ 10⁻⁴; the generator fails loudly if not). Distinguishing-move requirements may consume extra dates for `comboRush`/`longChainsOnly`.
- CI Flutter setup time (~2–4 min/run with caching) on the free tier.

## Out of scope

- Removing the dead `merge()` / `MergeEvent` code path, moving the continue-cap guard into `grantAdReward()`, extending `VerifyResult` — production cleanups for candidate #2's PR.
- The deploy-drift guard (detecting `_shared/` changes without a `submit-score` redeploy) — user chose minimal CI (option 1); needs Supabase auth as a repo secret. Future enhancement.
- Candidate #2 (moving the refill loop into `GameEngine`) — this fixture is its prerequisite, not its implementation.
- Any change to engine rules, scoring, or `kLeaderboardSeason`.
